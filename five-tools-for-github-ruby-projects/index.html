Five tools for Github Ruby projects

# 22 July 2015 on Ruby, Rails

Github starts as a project repository for teams. Now, it's a social network for developers and a platform to share and collaborate on projects.

Github, as its name says, it's an hub for Git projects. If you don't know Git, stop reading and learn it in 15 minutes. It's the tool changes my coder life.

Tools

Do you remember any badge on a Github repository? (see my Material Icon gem for an example) There are a lot of tools to integrate in your Ruby/Rails projects. In addition, some of them can be extended to other languages.

Some tools are paying services, but usually they are free for Open Source projects. Congratulations! Open your projects to collaborate has a prize (and not only one).

Travis CI

Testing is a very important stage of project development. It's the reason I go calm to home. But we must have in mind following idea:

Write tests is for humans and run them is for computers.

Travis CI perform the second job. Travis syncs Github projects and run tests when exist a new pull request or commit.

You can set multiple Ruby versions, install scripts, run sudo commands, set environment variables... everything you need to simulate a real context. If build fails, Travis will advice you.

Coveralls

Coveralls is a good tool to integrate with Travis CI. It reports you about the coverage of your tests. It means the number of relevant lines your tests cover.

The key of testing is to cover all important parts of your code.

Code climate

Do you deal with a bad code? Spend hours to understand it, unused classes and variables, memory leaks...

Code climate prevents us about bad practices in our code. It doesn't need any installation. Only login with your Github account and select the repo to inspect, a list of issues will appear after finish.

Hakiri

I know is difficult to secure our Ruby code. Every day, bugs are discovered and our apps are a combination of libraries (gems) and our code.

Hakiri helps us to keep safe our apps. It checks our code to find known bugs or bad practices and dependencies like gems.

If you think comments are unnecessary, remember please, you aren't a computer. We write a lot of lines, and of course, we will forget minor details. I recommend you to comment all projects even you are working alone.

A bad comment doesn't provide any new information:

# Iterate over elements
items.each do |item|
# ...
end

A good comment includes extra information you can't read in code:

# Format items to graph.js library
items.each do |item|
# ...
end

Inch CI is an interesting tool to be aware of how are our code is commented. It scans a repository and give us a mark for our project files.

Gemnasium (bonus tool from reddit)

Gemnasium is the best inspector for your project dependencies. It's complements hakiri due to it checks status of gems versions. Obsolete dependencies will set as "dangerous", and if they present some CVE it will turn to critical.
